# 📡 Tech Slop Bot

Личный агрегатор новостей из Telegram-каналов. Бот парсит публичные каналы, приводит тексты к единому редакционному формату через LLM (MiMo-V2.5-Pro) и присылает готовую ленту тебе в личку.

## Как это работает

```
cron-job.org (раз в N минут)
        ↓
дёргает GitHub Actions API (workflow_dispatch на bot.yml)
        ↓
запускает bot.py
        ↓
парсит t.me/s/channel для каждого добавленного канала
        ↓
новый текст → MiMo-V2.5-Pro по standardization_prompt.txt
        ↓
получает headline + bullets + hashtag (одна новость или несколько, если пост-дайджест)
        ↓
форматирует и отправляет сообщения в Telegram
        ↓
коммитит обновлённый channels.json обратно в репозиторий
```

GitHub Actions сам по себе не запускается по расписанию надёжно (известное ограничение `schedule` для воркфлоу) — поэтому реальный таймер вынесен на внешний сервис **cron-job.org**, который дёргает `workflow_dispatch` через GitHub API.

## Управление через бота

| Команда | Действие |
|---------|----------|
| `/add @channel` | Добавить канал |
| `/remove @channel` | Удалить канал |
| `/list` | Список каналов (с названиями) |
| `/help` или `/start` | Помощь |

Бот реагирует только на сообщения от владельца (`CHAT_ID`), команды от других людей игнорируются.

## Формат сообщений

Каждая новость приходит в стандартизированном виде:

```
<жирный заголовок одним предложением>
- буллит с деталью (если их 2-4 — иначе buллетов нет)
- ещё буллит

🔗 публикация Название канала
#хэштег_категории
```

Если в исходном тексте была единственная дополнительная деталь (а не 2+) — она дописывается обычным текстом после жирного заголовка, отдельного буллита из одного пункта не бывает.

### Категории (hashtag)

Один тег на новость, по приоритету:

1. **ИИ** → `ии_модели_и_агенты`, `ии_инфраструктура`, `ии_внедрение`, `ии_рынок`, `ии_регулирование` (кроме агентной коммерции — она уходит в платформы)
2. **Кванты / кибербез / финтех / телеком** → соответствующий тег
3. **Платформы** → `платформы_экономика_платформ`, `платформы_экосистемы`, `платформы_агентная_коммерция`, `платформы_регулирование`
4. **ИТ-сектор** (широкая категория, всё технологическое что не попало выше) → `ит_сектор`
5. Если вообще ничего не подошло → `другое` (редкое исключение, не дефолт)

## Что пересылается

- Текст с сохранением форматирования (жирный, курсив, ссылки) и стандартизацией через LLM
- Гиперссылка на источник внутри заголовка — если она была в исходном посте
- Фото (одно или альбом); если итоговый текст длиннее 1024 символов (лимит Telegram на подпись к фото) — фото уходит без подписи, текст отдельным сообщением следом
- Посты с видео/документами без текста — приходят уведомлением со ссылкой на оригинал, сам файл не пересылается

## Ограничения

- Каналы должны быть **публичными** и иметь рабочее веб-превью (`t.me/s/channel`)
- Каналы с отключённым превью не поддерживаются — спарсить нечего
- Картинки приходят в сжатом качестве (ограничение веб-версии Telegram, оригиналы недоступны без юзербота)
- Видео и документы не пересылаются — только уведомление со ссылкой
- Стандартизация платная (MiMo API) — при исчерпании баланса или сбое API бот один раз присылает предупреждение и дальше работает в режиме fallback (сырой текст без обработки), пока проблема не устранится

## Настройка

### 1. GitHub Secrets

| Секрет | Описание |
|--------|----------|
| `BOT_TOKEN` | Токен бота от @BotFather |
| `CHAT_ID` | Твой Telegram ID (узнать у @userinfobot) |
| `MI_API` | Ключ API MiMo (`sk-...`, от api.xiaomimimo.com) |
| `GH_PAT` | Fine-grained token с правом **Contents: Read and Write** на репо — нужен воркфлоу самому, чтобы коммитить `channels.json` |

### 2. cron-job.org

Отдельный сервис, который запускает воркфлоу по расписанию (GitHub `schedule` ненадёжен для новых репо). Использует **свой** Fine-grained token, отдельный от `GH_PAT` — с правом только **Actions: Write**, без доступа к коду репозитория:

- **URL:** `https://api.github.com/repos/USERNAME/REPO/actions/workflows/bot.yml/dispatches`
- **Метод:** POST
- **Заголовки:**
  - `Authorization: Bearer ТОКЕН` (свой, Actions: Write only)
  - `Accept: application/vnd.github.v3+json`
  - `Content-Type: application/json`
- **Тело:** `{"ref":"main"}`
- **Расписание:** рекомендуется не чаще раза в 2-3 минуты — слишком частые запуски при долгой обработке через MiMo приводят к гонке при коммите состояния (хотя `concurrency` в `bot.yml` и retry с `git pull --rebase` уже защищают от потери данных)

### 3. channels.json

Хранит список каналов, их отображаемые имена и ID последнего обработанного поста:

```json
{
  "offset": 0,
  "channels": {
    "bbcrussian": {"name": "BBC News Русская служба", "last_seen": 1234}
  },
  "mimo_alert_sent": false
}
```

Старый плоский формат (`"channel": 1234`) мигрирует автоматически при первом запуске, имя подтягивается само в фоне.

### 4. standardization_prompt.txt

Системный промпт для MiMo — описывает формат вывода (headline/bullets/hashtag), правила стиля (кавычки-ёлочки, сокращения, тире), классификацию по тегам, обработку ссылок из исходного текста и крайние случаи (бессвязный текст, реклама, поздравления). Редактируется отдельно от кода.
